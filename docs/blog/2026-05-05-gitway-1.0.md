# Gitway 1.0 — pure-Rust SSH for Git

`gitway = "1.0.0"` is on crates.io, and signed binaries for Linux,
macOS, and Windows are on the
[GitHub Release page](https://github.com/Steelbore/Gitway/releases/tag/v1.0.0).
This is the first stable release of Gitway, the pure-Rust SSH
toolkit for Git: transport, keys, signing, and agent.

## Why?

General-purpose SSH clients (`ssh`, PuTTY, Pageant) carry complexity
that Git doesn't need — interactive shells, tunneling, agent
forwarding, hundreds of config directives.  That complexity causes
three concrete pain points for Git users:

- **Configuration errors** — a misconfigured `~/.ssh/config`
  silently routes traffic through the wrong key.
- **Fragile host-key trust** — the first-connection TOFU model
  forces developers to blindly accept a fingerprint.
- **Windows inconsistency** — multiple competing SSH
  implementations with incompatible agent protocols.

Gitway is opinionated: it pins host fingerprints for known
providers (GitHub, GitLab, Codeberg) at build time, ships a
focused subset of `ssh_config(5)` parsing, and presents a
consistent JSON contract so agents and CI can drive it
reliably.

## What's in 1.0

The four core gaps Gitway closes for Git users:

### 1. Drop-in `core.sshCommand`

```sh
gitway --install     # registers gitway as git's transport globally
git fetch            # uses gitway under the hood
```

No interactive shell, no PTY, no agent forwarding — Gitway
implements the SSH surface Git actually needs and stops there.

### 2. Real `~/.ssh/config` resolution

Day-to-day directives are honored end-to-end: `Host`, `HostName`,
`Port`, `User`, `IdentityFile`, `IdentitiesOnly`, `IdentityAgent`,
`CertificateFile`, `Include`, `StrictHostKeyChecking`,
`UserKnownHostsFile`, `ProxyCommand` (including the `=none`
disable sentinel), `ProxyJump`, `KexAlgorithms`, `Ciphers`,
`MACs`, `HostKeyAlgorithms`, `ConnectTimeout`,
`ConnectionAttempts`.  And `gitway config show <host>` mirrors
`ssh -G` for inspection:

```sh
gitway config show github.com           # human, OpenSSH-style
gitway config show github.com --json    # frozen schema_version="1.0.0"
```

`Match` blocks are parsed but never match in 1.0; full
evaluation is on the v1.1 roadmap.

### 3. `-vvv` debug parity with OpenSSH

```sh
gitway -vvv --test ghe.example.com 2>&1 | rg 'CAT_KEX|CAT_AUTH|CAT_RETRY'
gitway -vvv --debug-format=json --debug-categories=kex,auth -- \
    git fetch
```

Every host-key check, every authentication attempt, every applied
`~/.ssh/config` directive (with `(file, line, directive, value)`),
and every ProxyJump hop emits a structured tracing event.  JSONL
mode means CI log shippers can index records cleanly.

### 4. Connection retry, backoff, and timeouts

Network blips happen.  Gitway 1.0 wraps `connect()` with an
exponential-backoff loop with jitter, a fatal-vs-transient
classifier (auth and host-key errors are *fatal* — they never
retry), and per-attempt timeouts:

```sh
gitway --connect-timeout 5 --attempts 3 --max-retry-window 30 -- \
    git fetch
```

Per-attempt history surfaces in the `--test --json` envelope under
`data.retry_attempts` for observability.

## Bonus: pinned fingerprints, algorithm denylist, `@revoked`

The defenses you'd hope your SSH transport already had:

- **Pinned host-key fingerprints** for GitHub, GitLab, and
  Codeberg embedded at build time.  No TOFU for known providers.
- **Algorithm denylist** that unconditionally refuses DSA, 3DES,
  RC4, hmac-sha1-96, and the SSH-1 protocol — no flag re-enables
  them.
- **`@revoked` known_hosts entries** checked *before* any
  `StrictHostKeyChecking=no` bypass; revocations cannot be
  overridden by policy.

And the new `gitway hosts {add, revoke, list}` family for
managing your own user `known_hosts` file with HMAC-SHA1 hashed
entries (matching OpenSSH's `HashKnownHosts yes`).

## Pure Rust, all the way down

- Project-owned crates declare `#![forbid(unsafe_code)]`.
- Transport crypto via [russh](https://github.com/warp-tech/russh)
  with the `aws-lc-rs` backend (post-quantum-ready, no CMake on
  non-FIPS).
- Keys + signing via [`ssh-key`](https://github.com/RustCrypto/SSH)
  (RustCrypto stack: `ed25519-dalek 2.x`, `rsa 0.9`, p256/384/521).
- Stripped musl-x86_64 binary under 12 MB.
- MSRV pinned at Rust 1.88.

The library lives in [Steelbore/Anvil](https://github.com/Steelbore/Anvil)
and is published as
[`anvil-ssh = "1.0.0"`](https://crates.io/crates/anvil-ssh) so
you can use the same SSH stack programmatically.

## Stable JSON envelope contract

Every `--json` output and the always-JSON `gitway schema` /
`gitway describe` carry `metadata.schema_version = "1.0.0"`.
Agents, MCP discovery layers, and CI parsers can pin against it.
Patch bumps add no symbols; minor bumps add only.

```json
{
  "metadata": {
    "tool": "gitway",
    "schema_version": "1.0.0",
    "version": "1.0.0",
    "command": "gitway --test --host github.com",
    "timestamp": "2026-05-05T08:21:21Z"
  },
  "data": {
    "host": "github.com",
    "host_key_verified": true,
    "fingerprint": "SHA256:...",
    "authenticated": true,
    "retry_attempts": []
  }
}
```

Auto-JSON-mode is selected on `AI_AGENT=1`, `AGENT=1`, `CI=true`,
plus `CLAUDECODE=1`, `CURSOR_AGENT=1`, `GEMINI_CLI=1`.

## What's not in 1.0

Three deferrals worth calling out explicitly:

- **FIDO2 / `sk-ssh-*` hardware-backed keys.**  Pulled from 1.0
  to give us a clean release window without the hardware-test
  matrix.  Tracked for v1.1 in PRD §8.1.  Users on YubiKey,
  SoloKeys, or OnlyKey: stay on OpenSSH for transport until then.
- **Live `@cert-authority` host validation during KEX.**  The
  parsing, audit-log surfacing, and `@revoked` enforcement
  shipped in 1.0; live validation against a CA-signed server
  certificate during the handshake is blocked on russh
  upstream cert-host-key support.  When russh exposes the
  certificate to `check_server_key`, the validation lands in
  v1.1.
- **Full `Match` block semantics in `ssh_config`.**  Parsed but
  not evaluated.  `Match exec` materially changes the security
  model and needs a careful design pass.

## Get it

```sh
# Cargo
cargo install gitway

# Or: grab a binary
# Linux:   gitway-v1.0.0-linux-x86_64.tar.gz   (musl, fully static)
# macOS:   gitway-v1.0.0-macos-arm64.tar.gz
# Windows: gitway-v1.0.0-windows-x86_64.zip
# https://github.com/Steelbore/Gitway/releases/tag/v1.0.0

# Wire it up to git
gitway --install
```

Migration notes for v0.9 users: see
[`docs/migration-from-v0.9.md`](https://github.com/Steelbore/Gitway/blob/main/docs/migration-from-v0.9.md).
