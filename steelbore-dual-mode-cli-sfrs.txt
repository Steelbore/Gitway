1. Executive Summary
This Software Feature Requirement Specification (SFRS) defines the architecture, behaviors, and compliance requirements for implementing dual-mode, self-documenting command-line interfaces across all Project Steelbore CLI tools. The specification mandates that every CLI binary ships with two co-equal output personalities: a human-oriented mode optimized for interactive terminal sessions with color, tables, and TUI support; and an agent-native machine-readable mode optimized for consumption by LLM-based AI coding agents, automation pipelines, and structured-data shells.
The specification synthesizes findings from three independent research reports commissioned for this effort, community best practices from the agent-friendly CLI movement (2024–2026), and Ugo Enyioha’s eight canonical rules for writing CLI tools that AI agents can reliably operate. It encodes non-negotiable Steelbore Standard compliance requirements including ISO 8601 timestamps in UTC, UTF-8 encoding throughout, POSIX-first design philosophy, and the Steelbore six-token color palette.
The primary reference implementation is Anthropic’s ant CLI, which auto-detects TTY state to switch between pretty-printed JSON for terminals, compact JSON for pipes, and supports --format yaml, --format jsonl, --format explore (TUI) modes. This specification generalizes that pattern into a portable, shell-agnostic framework suitable for all Steelbore projects.

2.3. Non-Negotiable Conventions
The following conventions are mandatory across all modes and all shell targets. Violation of any item constitutes a blocking defect:
1. ISO 8601 + UTC. All timestamps MUST be formatted as ISO 8601 with explicit UTC timezone designator (Z suffix). Example: 2026-04-10T14:30:00Z. Local-time output is forbidden. The --local-time flag is explicitly prohibited. Duration values use ISO 8601 duration format (e.g., PT1H30M).
2. UTF-8 without BOM. All text output (stdout, stderr, file writes) MUST be encoded as UTF-8 without byte-order mark. The tool MUST NOT rely on system locale for encoding decisions. On Windows, the tool MUST set the console output code page to 65001 (UTF-8) at startup.
3. POSIX-first design. The default (no-flag) output mode MUST produce text that is correct and parseable in a POSIX sh environment using only POSIX-standard utilities (grep, awk, cut, sed, tr). No Bash-isms, no Nushell-isms, no PowerShell-isms in default output.
4. Metric and 24-hour. All measurements use SI/metric units. All time-of-day values use 24-hour format. No AM/PM. No imperial units.
5. GPL-3.0-or-later licensing. SPDX license identifier headers in all source files.


3. The Eight Rules of Agent-Friendly CLI Design
The following eight rules, derived from Ugo Enyioha’s “Writing CLI Tools That AI Agents Actually Want to Use” (February 2026), form the normative behavioral requirements for every Steelbore CLI tool. Each rule is stated, then decomposed into specific testable requirements.


3.1. Rule 1: Structured Output Is Not Optional
Statement: Every command that returns data MUST support a machine-readable structured output mode. Unstructured text-only output is acceptable only as the human-oriented default, never as the sole format.
Specific requirements:
* Every data-returning sub-command MUST accept --json to emit a single, complete JSON document to stdout.
* The JSON output MUST contain a superset of the human output: all fields visible in text mode plus additional machine-useful fields (raw IDs, full timestamps, metadata, pagination tokens).
* The JSON schema for each command MUST be stable across minor versions. Breaking schema changes require a major version bump and a deprecation cycle.
* Additionally, MUST support --format <fmt> where <fmt> is one of: json (default structured), jsonl (newline-delimited JSON for streaming), yaml (YAML 1.2), csv (RFC 4180), explore (TUI browser; see Section 5.3).
* When stdout is not a TTY and no explicit --format is given, the tool SHOULD auto-select json mode (following the ant CLI pattern).


3.2. Rule 2: Exit Codes Are the Agent’s Control Flow
Statement: Exit codes are the primary mechanism by which an AI agent determines success, failure type, and retry strategy. They MUST be meaningful, consistent, and documented.
Required exit code map:
Code
	Meaning
	Agent Behavior
	0
	Success
	Proceed; parse stdout
	1
	General failure
	Log error; may retry with different parameters
	2
	Usage error (bad arguments)
	Do not retry; fix invocation syntax
	3
	Resource not found
	Adjust target; do not retry same query
	4
	Permission denied
	Escalate to user or acquire credentials
	5
	Conflict (e.g., resource already exists)
	Use idempotent alternative or resolve conflict
	126
	Command not executable (POSIX reserved)
	Check PATH and permissions
	127
	Command not found (POSIX reserved)
	Install dependency or check PATH
	128+N
	Fatal signal N (POSIX reserved)
	Log crash; do not retry
	

* Exit codes 6–125 are available for tool-specific semantics and MUST be documented in the tool’s schema introspection output.
* In --json mode, a non-zero exit MUST also emit a structured error object to stderr (see Rule 5).


3.3. Rule 3: Make Commands Idempotent
Statement: Agents retry on failure. Commands that mutate state MUST be safe to re-execute without causing duplicate side effects or corruption.
* State-mutating commands MUST be idempotent by default. Running the same create command twice MUST NOT create a duplicate resource; it MUST return success with the existing resource’s data, or return exit code 5 (conflict) with the existing resource in the error payload.
* Prefer declarative verbs (ensure, apply, sync) over imperative verbs (create, delete) where feasible.
* Where true idempotency is impossible (e.g., sending an email), the tool MUST support --dry-run which outputs the planned action as structured data without executing it.
* The --dry-run flag MUST be supported on every command that performs a write, delete, or destructive operation.


3.4. Rule 4: Self-Documenting Beats External Docs
Statement: An agent should never need to read a man page, website, or PDF to understand how to use a command. The CLI must describe itself completely at runtime.
* Every tool MUST implement a schema sub-command (e.g., mytool schema <command>) that emits the full JSON Schema definition of the command’s accepted input parameters and output structure.
* The schema output MUST include: command name, description, parameters (name, type, required, default, enum values, description), output schema (JSON Schema for stdout in --json mode), exit codes and their meanings, and examples.
* The --help output MUST include at least two realistic usage examples per sub-command, including one demonstrating --json mode.
* Every tool MUST ship CLAUDE.md, AGENTS.md, and SKILL.md context files following the Steelbore Standard format, placed at the repository root.
* The tool MUST support mytool describe which emits a machine-readable capability manifest listing all sub-commands, their schemas, and supported output formats.


3.5. Rule 5: Structured Errors, Not Narrative Messages
Statement: When a command fails, the error output must be as structured and parseable as success output. Narrative error messages waste tokens and defy programmatic handling.
In --json mode (or when stdout is not a TTY), all errors MUST be emitted to stderr as a JSON object with the following schema:
{
  "error": {
    "code": "NOT_FOUND",
    "exit_code": 3,
    "message": "Repository 'foo/bar' does not exist",
    "hint": "Run 'mytool repo list --json' to see available repositories",
    "timestamp": "2026-04-10T14:30:00Z",
    "command": "mytool repo get foo/bar",
    "docs_url": "https://steelbore.dev/docs/repo-get"
  }
}


* The hint field implements the “tips thinking” pattern: it MUST contain the exact command syntax required to resolve or investigate the error, enabling the agent to self-correct without hallucination.
* The error.code field MUST be a stable, upper-snake-case string enum (e.g., NOT_FOUND, PERMISSION_DENIED, INVALID_ARGUMENT, CONFLICT, RATE_LIMITED, INTERNAL_ERROR).
* In human mode (TTY), the tool SHOULD still print the hint to stderr as colored text, but MAY use prose formatting.


3.6. Rule 6: Design for Composability
Statement: Every CLI tool is a node in a pipeline. Output must be designed so that downstream consumers (pipes, jq, from json, ConvertFrom-Json, or another CLI tool) can consume it without transformation.
* stdout MUST carry only data payload. Never mix progress indicators, warnings, banners, or decorative output into stdout.
* stderr MUST carry all diagnostic, progress, and error messages.
* When emitting JSON, the output MUST be a single valid JSON document (array or object). No leading/trailing text, no log lines interspersed, no ANSI escape codes.
* For streaming use cases, the --format jsonl mode MUST emit one valid JSON object per line, terminated by a newline character (0x0A). Each line MUST be independently parseable.
* Support --fields <field1,field2,...> to limit output to specific fields, reducing payload size for agents operating under token-budget constraints.


3.7. Rule 7: Consistent Noun-Verb Command Structure
Statement: Agents discover capabilities by exploring command trees. A predictable structure reduces the exploration search space and prevents hallucinated sub-commands.
* All Steelbore CLIs MUST use a consistent <tool> <noun> <verb> command hierarchy (e.g., ferrocast module list, caliper trace run, craton package install).
* The noun MUST always be singular (e.g., ferrocast module list, not ferrocast modules list).
* Standard verbs MUST be consistent across all tools: list, get, create, update, delete, apply, sync, describe, schema.
* Aliases for common verbs are permitted (e.g., ls for list, rm for delete) but MUST NOT appear in --json output or schema introspection; they are a human convenience only.
* Global flags (--json, --format, --fields, --dry-run, --verbose, --quiet, --no-color, --help, --version) MUST be identical in name, type, and behavior across all Steelbore tools.


3.8. Rule 8: Understand When MCP Beats CLI
Statement: CLI and MCP are complementary, not competing, surfaces. A tool should expose both when the operational context demands it.
* Every Steelbore CLI that exposes more than 10 sub-commands SHOULD also ship an MCP server surface, activated via <tool> mcp [--transport stdio|sse] following the Google gws pattern.
* The MCP server MUST derive its tool definitions directly from the same schema definitions used by the schema sub-command — one source of truth, two interfaces.
* CLI remains the primary interface for local development, CI/CD, and terminal-based agents (Claude Code, Codex CLI). MCP is the preferred interface for non-terminal agents, multi-tenant platforms, and enterprise control planes.
* The MCP tool definitions MUST use lazy loading (deferred schema injection) to avoid consuming the agent’s context window with tool definitions it may never invoke.


4. Output Mode Architecture
4.1. Mode Detection and Selection
The output mode is determined by a strict precedence cascade. The first matching condition wins:
6. Explicit flag. If --format <fmt> or --json is passed, use that mode unconditionally. --json is equivalent to --format json.
7. Environment variable. If AI_AGENT=1, AGENT=1, or CI=true is set, default to json mode with no color, no interactivity, and no TUI.
8. TTY detection. If stdout is connected to a TTY (isatty(stdout) returns true), use human mode with color, formatted tables, and optional TUI. If stdout is not a TTY (piped or redirected), default to json mode.
9. Fallback. If none of the above conditions apply, use human mode.


4.2. Human-Oriented Mode (Default Interactive)
When operating in human mode, the tool presents output optimized for visual consumption in a terminal emulator:
* Colored text using the Steelbore six-token palette, applied via ANSI escape sequences. Success messages use Radium Green (#50FA7B), warnings use Molten Amber (#D98E32), errors use Red Oxide (#FF5C5C), informational text uses Steel Blue (#4B7EB0), data values use Liquid Coolant (#8BE9FD).
* Column-aligned tabular output for list commands, using Unicode box-drawing characters for borders. Tables MUST be grep-parseable by keeping data on single lines where feasible.
* Relative timestamps for recency (e.g., “3 minutes ago”) with absolute ISO 8601 UTC available via --absolute-time flag. Note: the underlying data is always stored and transmitted as ISO 8601 UTC; relative display is a rendering convenience only.
* Progress indicators (spinners, progress bars) rendered to stderr only, never to stdout.
* Interactive prompts (confirmations, selection menus) rendered to stderr. When stdin is not a TTY, all prompts MUST auto-decline (safe default) and emit a structured error to stderr explaining which flags are required for non-interactive invocation (the “Wizard Fallback” pattern).


4.3. Machine-Readable Mode (--json / --format)
When operating in machine mode, the tool produces output optimized for programmatic consumption:
* JSON output MUST be a single, complete, valid JSON document. No trailing commas, no comments, no BOM, no ANSI escape codes.
* All timestamps as ISO 8601 with Z suffix. No relative times, no locale-dependent formatting.
* All numeric values as JSON numbers (not strings). Booleans as true/false (not "yes"/"no").
* Null values as JSON null (not empty strings, not "N/A").
* Property names in snake_case for all Steelbore projects. PascalCase aliases MAY be provided for PowerShell friendliness but snake_case is canonical.
* The top-level JSON object MUST include a metadata wrapper with the tool version, command invoked, timestamp, and pagination info:
{
  "metadata": {
    "tool": "caliper",
    "version": "0.3.0",
    "command": "caliper trace list",
    "timestamp": "2026-04-10T14:30:00Z",
    "pagination": { "page": 1, "per_page": 50, "total": 142, "next_token": "abc123" }
  },
  "data": [ ... ]
}


4.4. Color and Formatting Control
Color output follows a strict precedence chain compatible with the NO_COLOR community standard and the CLICOLOR convention:
10. --color=never / --no-color flag: Disable color unconditionally.
11. --color=always flag: Enable color unconditionally (including when piped).
12. NO_COLOR environment variable: If set and non-empty, disable color. Takes precedence over CLICOLOR.
13. FORCE_COLOR environment variable: If set and non-empty, enable color (even when piped). Overrides NO_COLOR.
14. CLICOLOR=0: Disable color.
15. TTY detection: If stdout is a TTY, enable color. Otherwise, disable.


* In machine mode (--json, --format json/yaml/csv/jsonl), all ANSI escape codes MUST be suppressed regardless of color settings.
* In --format explore (TUI mode), full color is used unless NO_COLOR is set.


5. TUI Mode Specification (--format explore)
5.1. Purpose and Activation
The explore mode launches a Terminal User Interface (TUI) for interactively browsing, filtering, and navigating structured output data. This mode is inspired by Anthropic’s ant CLI --format explore and Nushell’s explore command. It provides a human-optimized data exploration experience while maintaining the same underlying data model as --json.
* Activated via: --format explore or the short alias -E.
* Requires stdout to be a TTY. If stdout is not a TTY, the tool MUST fall back to --format json and emit a warning to stderr.
* Graceful degradation: If the AI_AGENT=1 or AGENT=1 environment variable is set, TUI mode MUST NOT activate. The tool MUST fall back to --format json and emit the full data payload. This prevents agents from being trapped in an interactive render loop.


5.2. TUI Behavioral Requirements
* The TUI MUST support both CUA-standard keybindings (arrow keys, Page Up/Down, Home/End, Tab) and Vim-style keybindings (hjkl navigation, /, G, gg) simultaneously, per the Steelbore Standard dual-keybinding requirement.
* Rendering MUST use the Steelbore six-token color palette exclusively: Void Navy for backgrounds, Molten Amber for data values, Steel Blue for borders and headers, Radium Green for selected/active items, Red Oxide for errors, Liquid Coolant for metadata and types.
* The TUI MUST render to an alternate screen buffer so that the user’s previous terminal content is preserved upon exit.
* All TUI rendering MUST go through stderr or the alternate screen buffer. The final selected/exported data (if any) MUST be emitted to stdout as JSON upon exit, enabling pipelines like: mytool list -E | jq '.name'
* The TUI MUST support inline search/filter (press / to filter table rows by substring match).
* The TUI MUST support column sorting (press s to cycle sort by current column).
* The TUI MUST support detail view (press Enter on a row to expand a full record view with all fields, including those hidden from the summary table).
* The TUI MUST support export (press e to write the current filtered/sorted view as JSON to a file).


5.3. TUI Implementation Constraints
* The TUI framework MUST be pure Rust (no C dependencies). Recommended: ratatui with crossterm backend.
* The TUI MUST NOT use raw terminal mode in CI or non-TTY contexts. The isatty check is mandatory before entering raw mode.
* The TUI binary size overhead MUST be gated behind a Cargo feature flag (tui) so that headless/server deployments can build without TUI dependencies.


6. Schema Introspection and Self-Description
6.1. The schema Sub-Command
Every Steelbore CLI MUST implement <tool> schema [<noun> [<verb>]] which emits the JSON Schema for the specified command’s input and output. When invoked without arguments, it MUST emit the complete tool schema covering all commands.
The schema output MUST conform to JSON Schema Draft 2020-12 and include:
* Input parameters: name, type, required, default, enum values, description, and constraints (e.g., minimum, maximum, pattern).
* Output schema: the JSON Schema of the data field in the response envelope when --json is used.
* Exit codes: a map of exit code integers to human-readable descriptions.
* Examples: at least one example invocation with expected output per command.
* The schema output MUST be directly usable as an LLM function-calling tool definition (i.e., compatible with the tools[].function.parameters format used by OpenAI, Anthropic, and Google Gemini APIs).


6.2. The describe Sub-Command
Every Steelbore CLI MUST implement <tool> describe which emits a compact capability manifest:
{
  "tool": "caliper",
  "version": "0.3.0",
  "description": "Rust raster-to-vector tracing engine",
  "commands": [
    {
      "name": "caliper trace run",
      "description": "Execute a tracing pipeline on an input image",
      "supports_json": true,
      "supports_dry_run": true,
      "idempotent": false,
      "destructive": false
    }
  ],
  "global_flags": ["--json", "--format", "--fields", "--dry-run", ...],
  "output_formats": ["json", "jsonl", "yaml", "csv", "explore"],
  "mcp_available": true
}

7. Input Validation and Agent-Safety Hardening
7.1. Threat Model
When an AI agent invokes a CLI tool, the agent is not a trusted operator. Agents may hallucinate arguments, inject control characters, or pass adversarial payloads received from untrusted upstream data (indirect prompt injection). The CLI tool MUST defend against:
* Path traversal attacks (../../etc/passwd) in any argument accepting file paths.
* Control characters (0x00–0x1F except 0x0A and 0x0D) embedded in string arguments.
* Double-encoded UTF-8 sequences.
* Embedded shell metacharacters in arguments that will be interpolated into sub-processes.
* Unreasonably large input values designed to exhaust memory or disk.


7.2. Required Validations
16. All file path arguments MUST be canonicalized and validated against a configurable allowed-paths list. Paths outside the allowed set MUST be rejected with exit code 4 (PERMISSION_DENIED).
17. All string arguments MUST be validated for control characters. Arguments containing bytes 0x00–0x08, 0x0B–0x0C, or 0x0E–0x1F MUST be rejected with exit code 2 (USAGE_ERROR).
18. All numeric arguments MUST be bounds-checked against the parameter’s declared minimum/maximum constraints from the schema.
19. The --dry-run flag MUST be honored before any side effects occur. The dry-run output MUST include the full action plan as structured JSON.
20. Confirmation prompts for destructive operations MUST default to “no” in non-interactive contexts. The --yes / --force flag is required to skip confirmation in non-TTY mode.


8. Shell-Specific Output Requirements
8.1. POSIX / Bash / Brush Compatibility (P0)
These requirements ensure correct operation in the baseline text-stream shell environment:
* Default text output MUST use single-line records suitable for grep/awk/cut processing. Field separators MUST be consistent (tab-separated for structured data, space-aligned for human tables).
* Avoid multi-line field values in text mode. If a field contains newlines, truncate or replace with a Unicode ellipsis (…) in text mode; preserve in --json mode.
* NUL-delimited output MUST be available via --print0 or -0 for filename-containing output, enabling safe consumption via xargs -0 or while read -d ''.
* The tool MUST NOT use Bash-specific features in its own shell scripts or completions if targeting POSIX. Shell completion scripts MUST be provided for bash, zsh, fish, and nushell.
* Brush compatibility is automatic: Brush’s behavioral contract guarantees identical treatment of text streams. No Brush-specific accommodations are needed.


8.2. Nushell Integration (P1)
Nushell users consume CLI output primarily via the from json pipeline command. Requirements:
* The --json output MUST be a single valid JSON document (not NDJSON by default) because Nushell’s from json expects a complete document. Use --format jsonl for streaming.
* JSON property types MUST be semantically correct: numbers as JSON numbers (Nushell infers int vs float), dates as ISO 8601 strings (Nushell’s into datetime converts them), filesizes as integers in bytes (Nushell’s into filesize converts them), durations as ISO 8601 duration strings.
* Non-zero exit codes MUST correctly signal errors because Nushell (since v0.98) treats non-zero exits as errors by default.
* When outputting lists of homogeneous records (the common case for list commands), the JSON MUST be a top-level object with a data field containing an array of objects with consistent keys — this maps cleanly to Nushell’s table type.
* Steelbore CLI tools MAY optionally provide Nushell plugin wrappers in a separate nu-plugin-<tool> crate, but this is not required; --json plus from json is the primary integration path.


8.3. PowerShell 7+ Integration (P1)
PowerShell users consume CLI output via ConvertFrom-Json. Requirements:
* The --json output MUST be a single JSON document (not NDJSON). PowerShell’s ConvertFrom-Json expects a complete JSON string, not streaming lines.
* JSON nesting MUST NOT exceed depth 3 without strong justification. PowerShell’s ConvertTo-Json defaults to depth 2 for round-tripping; ConvertFrom-Json supports depth 1024 for reading, but shallow structures are preferred.
* Property names SHOULD use snake_case as canonical but the tool MAY additionally provide PascalCase aliases in the output for PowerShell ergonomics (e.g., both created_at and CreatedAt).
* UTF-8 without BOM is mandatory. The tool MUST NOT emit UTF-16 or rely on the system’s OEM code page.
* Errors on stderr MUST be single-line JSON objects (not multi-line) because PowerShell wraps each stderr line in a separate ErrorRecord. Multi-line stderr JSON is fragmented and unparseable in PowerShell.


8.4. Ion Shell (RedoxOS) Compatibility (P2)
Ion is a text-stream shell with typed variables but no structured pipeline. Requirements:
* Default text output works with Ion identically to POSIX sh. No special accommodations required.
* Ion’s improved stderr redirection (^>, ^|, &>, &|) means that tools properly separating stdout and stderr will benefit from more granular redirection in Ion. This reinforces the requirement for strict stdout/stderr separation.
* Ion’s native array support (@(command) for array splitting) means that list output should have one record per line for clean array capture in Ion scripts.
* JSON parsing in Ion requires external tooling (jq). The --json mode requires no special adaptation.

10. MCP Server Surface
10.1. Activation
Steelbore CLI tools that expose more than 10 sub-commands SHOULD implement an MCP server mode, activated via:
<tool> mcp [--transport stdio|sse|streamable-http]


* The default transport MUST be stdio (JSON-RPC over stdin/stdout), which is the most universally supported transport for local MCP servers.
* The MCP server MUST derive its tool definitions from the same schema sub-command output, ensuring one source of truth.


10.2. Lazy Schema Loading
To mitigate the MCP context-window overhead problem (schema bloat consuming 35× more tokens than equivalent CLI invocations), the MCP server MUST implement deferred tool schema loading:
* On connection, the server MUST advertise only tool names and one-line descriptions.
* Full input/output schemas are loaded on demand when the agent selects a specific tool (the tools/get pattern).
* This follows the lazy-loading pattern demonstrated by Claude Code’s deferred MCP schema injection and Cursor’s tool-search approach (46.9% token reduction).


11. Testing and Compliance Verification
11.1. Required Test Categories
Every Steelbore CLI MUST include the following test categories in its CI pipeline:
21. JSON Schema Validation Tests. For every command that supports --json, a test MUST validate that the actual output conforms to the declared schema from the schema sub-command.
22. Exit Code Tests. Every documented exit code (0–5 plus tool-specific codes) MUST have at least one test case that triggers it and verifies the correct code.
23. TTY/Non-TTY Mode Tests. Tests MUST verify that output format changes correctly when stdout is/is not a TTY.
24. Agent Environment Tests. Tests MUST verify correct behavior when AI_AGENT=1 is set: no color, no interactivity, JSON output.
25. Idempotency Tests. Every state-mutating command MUST have a test that runs the command twice and verifies no duplicate side effects.
26. Input Validation Tests. Path traversal, control character injection, and bounds-violation inputs MUST be tested for correct rejection.
27. Cross-Shell Roundtrip Tests. JSON output MUST be tested for correct parsing through jq (POSIX/Bash), from json (Nushell), and ConvertFrom-Json (PowerShell).
28. UTF-8 Encoding Tests. Output containing non-ASCII characters (Arabic, CJK, emoji) MUST be verified as valid UTF-8 without BOM.
